# Create: Pocket Factory 开发进程文档

## 1. 文档目标

本文档用于固化 Create: Pocket Factory 的阶段性开发方案，限制第一阶段范围，提前确定关键设计决策，避免在实现过程中因需求扩散导致返工。

当前目标不是一次性做完整的跨维度工厂系统，而是先做出一个可验证性能收益、可稳定传送、可长期扩展的原型。

## 2. 项目定位

Create: Pocket Factory 是一个面向机械动力自动化玩家的独立维度工厂模组。

核心思想：

1. 将高渲染负担的自动化机器迁移到单独维度。
2. 玩家离开工厂后，机器继续运行，但不再占用常驻活动区域的渲染预算。
3. 第一阶段优先解决“渲染隔离”和“稳定传送”，不提前承诺复杂自动化兼容。

非目标：

1. 第一阶段不解决所有服务端逻辑性能问题。
2. 第一阶段不做传送带、管道、溜槽等跨维度自动化兼容。
3. 第一阶段不做多工厂实例。
4. 第一阶段不做结构打包迁移。

## 3. 已确认的关键决策

以下决策已确定，后续开发默认以此为准：

1. 玩家从工厂返回时，优先回到进入前记录的位置；若原位置无效，再回主世界出生点。
2. 第二阶段开始使用“入口绑定工厂实例”的模型，不采用全局共用入口。
3. 工厂维度内不允许创建新的工厂入口，避免递归嵌套。
4. 第二阶段入口优先采用单方块，而不是多方块门结构。
5. 后续跨维度输入输出优先抽象为“独立绑定 + 共享后端/桥接后端”，绑定真值以 binding ID 为核心，而不是按玩家 ID 或工厂 ID 直接寻址。
6. 光照策略采用“工厂内部默认高亮环境 + 配置开关”，避免用侵入性过强的全局光照逻辑。
7. 常加载仅覆盖实际工厂区域，不扩大到整个维度。

## 4. 第一阶段范围定义

第一阶段目标是做出最小可玩的原型，并验证“把机器迁移到独立维度后，玩家日常活动区域的帧率是否改善”。

### 4.1 玩家可见功能

1. 新增一个手持物品，作为第一阶段唯一入口物品。
2. 物品贴图复用末影之眼，改为黄色方案。
3. 玩家在非工厂维度右键该物品时，传送到口袋工厂维度的固定工厂房间中心。
4. 玩家在工厂维度右键该物品时，返回进入前记录位置。
5. 若返回位置失效，则回主世界出生点。

### 4.2 世界与维度功能

1. 新增维度：口袋工厂。
2. 在固定坐标生成第一个工厂盒子。
3. 工厂盒子以不可破坏边界为基础实现。
4. 推荐实现：基岩地板 + 屏障墙与顶，或完整基岩盒子。
5. 工厂区域对应区块保持常加载。
6. 工厂维度不自然刷新生物。
7. 工厂内部提供默认高亮环境，并暴露配置开关。

### 4.3 第一阶段验收标准

1. 玩家可稳定往返工厂与原世界。
2. 重登后工厂与传送逻辑不丢失。
3. 工厂区块离开后仍保持机器运行。
4. 工厂维度内不自然刷怪。
5. 工厂内部亮度满足基本建造与观察需求。
6. 玩家常驻区域相比原先明显减少机器渲染压力。

## 5. 技术设计边界

### 5.1 传送与返回点

第一阶段必须记录玩家进入工厂前的位置数据。

建议记录内容：

1. 维度 ID。
2. 精确坐标。
3. 朝向。
4. 是否存在有效回传点。

返回流程建议：

1. 若玩家位于非工厂维度且使用入口物品，则记录当前位置并传送到工厂。
2. 若玩家位于工厂维度且使用入口物品，则尝试传送回记录点。
3. 若记录点所在维度不存在、区块不可用、目标位置危险或数据丢失，则回主世界出生点。

### 5.2 工厂盒子

第一阶段仅维护一个固定尺寸盒子。

建议：

1. 占地先固定为 1 个区块或 2 个区块，后续再扩展。
2. 高度可以先固定为较低但足够搭建原型机器的范围。
3. 中心区域预留安全传送点。
4. 墙体、顶部和底板的职责应拆分清楚，便于未来替换为可扩展外壳。

### 5.3 常加载

常加载只覆盖工厂盒子实际占用区块。

第一阶段重点验证：

1. 玩家离开后机器是否继续运作。
2. 服务器重启后强制加载状态是否恢复。
3. 常加载逻辑是否会被错误扩散到无关区块。

### 5.4 光照与刷怪

光照不建议采用会影响整个位面规则的激进方案，应优先使用以下思路：

1. 工厂建筑内部采用天然高亮的方块组合。
2. 通过维度规则或刷怪限制，阻止自然生物生成。
3. 若需要额外亮度强化，做成配置项而不是硬编码。

## 6. 阶段路线图

## 阶段 1：原型验证版

目标：

完成最小闭环，验证设计方向成立。

内容：

1. 单入口手持物品。
2. 单一口袋工厂维度。
3. 单个固定工厂盒子。
4. 进出传送。
5. 回传原位置。
6. 区块常加载。
7. 禁止自然刷怪。
8. 工厂内部高亮环境。

产出标准：

1. 能玩。
2. 稳定。
3. 有可感知的渲染收益。

## 阶段 2：固定入口版

目标：

将入口从手持物品切换为世界中的固定设施。

内容：

1. 新增单方块入口方块。
2. 右键入口进入工厂。
3. 工厂内生成对应回传核心或回传方块。
4. 为未来实例化绑定模型预留数据结构。

暂不做：

1. 多方块门结构。
2. 自动化接口。

当前已落地的第二阶段切片：

1. 新增固定入口方块 `pocket_factory_entrance`，作为正式入口原型。
2. 入口方块当前使用白灰棋盘格镂空框架外观，并在方块内部渲染一个旋转中的专用内部眼睛物品薄片。
3. 入口方块可在普通维度直接右键进入口袋工厂，复用第一阶段的返回点记录逻辑。
4. 工厂内部继续禁止使用入口方块，避免递归入口。
5. 工厂内的“口袋工厂方块B”现在可直接作为出口，右键即可返回记录位置或主世界出生点。
6. 手持入口物品已改为“创造模式口袋工厂调试之眼”，并与入口内部旋转眼睛拆分为独立资源。

建议按以下顺序落地第二阶段：

1. 新增固定入口方块及其物品、模型、方块状态和掉落表。

## 阶段 3：实例化工厂版

目标：

让一个世界支持多个工厂实例，并先完成入口实例与工厂实例的稳定绑定。

内容：

1. 每个工厂拥有唯一工厂 ID。
2. 每个入口绑定一个独立工厂。
3. 工厂具备所有者、名称和基础元数据。
4. 拆除入口后工厂数据继续保留，并允许后续恢复绑定。
5. 支持多人访问控制的基础框架。

注意：

1. 工厂维度内依旧禁止创建新工厂入口。
2. 需要明确实例存档格式，避免后期迁移困难。

### 阶段 3 当前分析（2026-05-15）

本阶段的核心不再是传送，而是“入口实例”和“工厂实例”的稳定绑定关系。

本阶段必须同时满足的约束：

1. 一个世界内允许存在多个入口方块。
2. 每个入口方块只绑定一个工厂实例。
3. 拆除入口方块后，工厂实例本身不能被删除。
4. 同一个入口方块重新放置后，应能恢复到原绑定工厂，而不是生成新工厂。
5. 玩家即使暂时失去入口方块，也不应该因为一次误拆而永久丢失工厂数据。
6. 每个新增工厂与其他工厂的边界必须保持足够远的水平距离，当前下限按至少 50 个方块处理。

推荐的数据归属方式：

1. 工厂实例数据存放在独立的世界级 `SavedData` 或等价持久化结构中，而不是仅挂在入口方块实体上。
2. 入口方块只保存“当前绑定的工厂 ID”和少量显示/同步信息。
3. 方块被拆除时，世界级工厂实例保留，仅将入口位置标记为未放置或失效。
4. 方块掉落物或拾取后的物品数据应携带工厂 ID，以支持重新放置后恢复绑定。
5. 若入口掉落物丢失，调试菜单仍应能列出已有工厂并允许恢复或重新生成入口。
6. 工厂注册表中应直接记录工厂中心点、占用边界和保留缓冲距离，避免后续生成与扩容时才临时推导。

推荐的第三阶段最小闭环：

1. 放置新的入口方块时，若物品中没有工厂 ID，则创建新工厂实例并写入物品/方块绑定。
2. 创建新工厂实例时，必须在工厂维度内选择一个与其他工厂边界至少间隔 50 方块的落位。
3. 放置已有绑定数据的入口方块时，直接恢复到原工厂实例。
4. 拆除入口方块时，不销毁工厂，仅将绑定入口位置清空并把工厂 ID 写回掉落物。
5. 进入工厂时优先读取入口绑定的工厂 ID，而不是全局唯一工厂。
6. 工厂内部出口返回逻辑仍沿用玩家返回点，不在第三阶段一开始就改动。

第三阶段需要优先规避的风险：

1. 方块复制、创造模式中取块、拾取失败等情况可能导致工厂 ID 被重复复制。
2. 工厂实例与入口位置不同步时，容易出现“入口已拆但仍显示已放置”的脏状态。
3. 玩家手动删除工厂维度内对应区域后，需要明确入口是否仍可恢复该工厂。
4. 多人环境下，入口所有权、共享访问和抢占恢复流程需要先限定规则再实现。
5. 工厂间距如果过近，会同时引入后续扩容碰撞风险和远视距下的额外渲染负担。

当前建议的实现顺序：

1. 先实现独立工厂注册表、工厂 ID 分配和最小间距选址。
2. 再实现入口方块与掉落物的数据绑定。
3. 再实现拆除后保留工厂、重放后恢复绑定。
4. 再实现基于工厂 ID 的进入逻辑与恢复入口流程。
5. 调试菜单和高层设置项延后到 3.5 阶段。

### 阶段 3 当前非目标

## 阶段 4：工厂扩容版

目标：

允许工厂尺寸增长。

内容：

1. 按整区块扩展长度与宽度。
2. 按整数高度扩展垂直空间。
3. 扩容时重建边界、照明和传送落点。
4. 为未来墙体接口留出规则化布局。
5. 扩容前必须检查与其他工厂的最小安全间距，避免扩容后占位碰撞。

### 阶段 4 当前规则（2026-05-15 确认，2026-05-16 收口）

#### 4.1 扩容触发物品

1. 新增正式物品“口袋工厂核心”。
2. “口袋工厂核心”与当前入口内部旋转的棕黄色眼睛使用同一个物品与同一套贴图资源。
3. 口袋工厂入口的配方使用白色羊毛位于四角、灰色羊毛位于四边，中间放置一个“口袋工厂核心”。
4. “口袋工厂核心”的配方使用泥土和木板围绕一周，中间放置一个末影珍珠。

#### 4.2 扩容交互入口

1. 只有玩家手持“口袋工厂核心”右键工厂边界方块 A 时，才会尝试触发扩容判定。
2. 若玩家未手持核心，则边界 A 不触发扩容。
3. 若点击位置不在口袋工厂维度内，或该方块不属于任何一个工厂的有效边界，则本次交互视为无效，回退到原有方块/物品交互语义。

#### 4.3 扩容方向判定

1. 玩家只能站在工厂内部与边界交互，因此水平方向的扩容判定采用“点击面的反方向”。
2. 例如玩家在工厂内部点击某边界方块的西侧面，实际扩容方向为东侧。
3. 垂直方向同样需要按“从工厂内部点击边界”的语义解释。
4. 玩家站在工厂内部点击脚下底部边界方块的顶部面时，实际应判定为“向下扩容”。
5. 玩家点击头顶顶部边界方块的底部面时，实际应判定为“向上扩容”。
6. 首版阶段 4 不做自动对角线扩容，也不做失败后的自动换向尝试。
7. 一次交互只能产生一个明确方向的扩容结果；若该方向非法，则直接失败并提示原因。

#### 4.4 横向与纵向扩容单位

1. 左右前后四个水平方向，每次只允许扩展 1 个完整区块。
2. 向上和向下扩容默认每次扩展 1 格高度。
3. 为避免游戏内交互语义混淆，阶段 4 收口时已移除原计划中的“潜行右键一次扩展 5 格”快捷扩容分支。
4. 阶段 4 继续保持“水平按区块扩、垂直按格子扩”的总体约束。

#### 4.5 核心消耗规则

1. 水平方向扩容成功时，消耗玩家手中的 1 个“口袋工厂核心”。
2. 向上或向下扩容成功时，不消耗核心。
3. 无论是否消耗核心，玩家都必须手持核心才能触发扩容判定，以避免误触顶部或底部边界时意外扩容。

#### 4.6 最大尺寸限制

1. 单个工厂在水平方向上的最大扩容半径限制为 30 个区块。
2. 设置该上限的原因是：在当前 1024 方块槽位间距下，需要为另一个同样扩到极限的工厂保留最低安全余量。
3. 若本次扩容会让工厂任一水平方向超出 30 区块半径，则必须直接失败。

#### 4.6.1 尺寸数据模型

1. 阶段 4 起，工厂的横向尺寸不再使用单一矩形宽高描述，而是改为记录“已占用区块集合”。
2. 每个工厂继续保留自身的基础槽位 `slotX/slotZ`，用于确定该工厂在口袋维度中的大坐标锚点。
3. 在此基础上，额外记录一个相对区块偏移集合 `(dx, dz)`，用于描述该工厂当前实际占用的所有区块。
4. 默认初始工厂始终包含基础区块 `(0, 0)`。
5. 水平方向后续新增的每一个区块，都作为一个独立的相对偏移条目写入工厂存档，而不是被压缩成统一的长宽范围。
6. 这样做的目的是允许玩家把工厂扩展成不规则平面形状，而不仅限于完整矩形。
7. 阶段 4 首版要求：新增加的水平区块必须与现有已占用区块集合四向相邻，以保持工厂平面整体连通。
8. 阶段 4 首版不支持删除已扩出的区块，只支持新增。
9. 垂直尺寸只记录两个整数值：`minY` 与 `maxY`。
10. 同一个工厂的所有已占用区块共享同一组 `minY/maxY`，阶段 4 首版不支持“不同区块拥有不同高度”的复杂垂直轮廓。

#### 4.7 初始基准高度调整

1. 为了支持向下扩容，阶段 4 起工厂初始盒子的整体基准高度需要上移，避免底部过于贴近 y=0。
2. 当前仍处于早期测试阶段，不需要为旧工厂存档提供兼容迁移。
3. 因此阶段 4 可以直接调整初始工厂布局与生成高度，并允许通过删除测试存档重新验证。

#### 4.8 阶段 4 首版验收重点

1. 水平方向扩容后，新增加的区块必须被纳入边界重建和常加载范围。
2. 垂直方向扩容后，顶部/底部边界必须正确重建，且不会破坏传送与出口逻辑。
3. 扩容失败时必须明确说明原因，例如未手持核心、点击位置无效、超出最大半径或目标方向不可扩展。
4. 阶段 4 首版先不处理旧工厂兼容，也不处理复杂图形扩容界面。

### 阶段 4 收尾记录（2026-05-16）

当前第四阶段已经完成首版收口，可作为后续跨维度接口阶段的稳定基线。

已完成并通过实测的能力：

1. 工厂尺寸模型已升级为“横向已占用区块集合 + 纵向 `minY/maxY`”，支持后续不规则平面扩展。
2. 玩家手持“口袋工厂核心”右键边界方块 A 时，可以稳定触发水平方向扩容，并在成功时消耗 1 个核心。
3. 垂直方向的顶部/底部边界重建、房间外壳更新、常加载范围更新，以及传送落点/出口逻辑，已在当前首版逻辑下保持闭环。

## 阶段 5：跨维度库存桥版

目标：

先做稳定、高性能、可复用的跨维度共享存储后端，再基于同一后端逐步挂接玩家容器、漏斗、传送带、管道、溜槽等不同访问端点。

阶段状态说明：

1. 本阶段早期文档曾采用“贴入口自动继承工厂 ID + 单工厂默认共享通道”的原型路线。
2. 该路线现已确认为历史过渡方案，不再作为长期架构继续扩展。
3. 保留以下阶段 5 内容的主要价值，是记录共享物品箱/共享流体罐的后端语义、容量规则、守恒约束和本地表现层原则。
4. 自本次修订起，所有“工厂 ID 直接作为共享后端主键”“贴入口自动绑定即长期绑定真相”“单工厂单默认通道”表述，均视为已废弃。

内容：

1. 本阶段不再让当前单方块工厂入口兼任共享容器入口。
2. 新增独立的“共享接口块”作为阶段 5 原型，用于验证端点表现层与世界级共享后端分离的路线。
3. 共享接口块的长期目标不是继承工厂绑定，而是接入独立 Binding Core，并通过 binding ID 访问同一份中心化后端数据。
4. 阶段 5 首版先打通玩家手动存取与漏斗等基础容器访问，再为后续自动化桥接预留统一后端。
5. 阶段 5 第一版共享原型拆分为“共享物品箱”和“共享流体罐”两类端点，不做单方块万能混合接口。

建议顺序：

1. 先做中心化共享存储后端。
2. 再做独立接口块与自动绑定原型。
3. 再做玩家 GUI 与漏斗等基础交互。
4. 最后才做传送带、管道、溜槽等 Create 生态专用接口。

### 阶段 5 当前方案（2026-05-16）

#### 5.1 核心目标

1. 阶段 5 的核心不是“再加一个箱子”，而是先建立一套世界级共享后端与端点表现层分离的容器语义。
2. 该后端必须足够轻量，避免为每个接口端点重复维护库存副本或执行持续扫描。
3. 绑定到同一个 binding ID 的多个同类端点，看到的物品内容或流体内容必须完全一致。
4. 后续不论是玩家手动打开、漏斗抽插，还是传送带桥、管道桥、溜槽桥，都必须复用同一后端，而不是各做一套存储实现。

#### 5.2 入口与接口职责拆分

1. 当前单方块入口的职责继续保持为“进入工厂 + 工厂绑定”，不再叠加共享库存 GUI。
2. 阶段 5 首版新增独立的共享接口块，作为外部世界和工厂内部的统一访问端点。
3. 共享接口块本身不拥有独立于后端的真实存储真相；真实物品与真实流体都统一保存在世界级共享存储注册表中。
4. 但共享接口块在绑定后仍必须保留完整的本地端点语义，用于向外暴露 Create 生态所期待的能力接口、控制器状态、液位显示与附属逻辑。
5. 对外部自动化与附属方块来说，绑定后的共享端点应继续表现为“一个正常可交互的本地容器端点”，只是其真实存储来源被切换为共享后端。
6. 这样做的直接收益是：入口的传送交互不会和容器交互冲突，后续入口升级为多方块结构时也无需推倒共享存储后端。
7. 入口不再被视为共享绑定流程中的必要提交器；它只是工厂侧的一种候选端点或一种工厂上下文来源，而不是所有绑定关系的唯一锚点。

#### 5.3 阶段 5 首版原型形态

1. 阶段 5 已完成的共享箱罐原型，可被视为“绑定真值尚未抽离前的共享后端验证切片”。
2. 它证明了世界级共享物品后端、世界级共享流体后端、本地控制器能力投影和结构显示同步这些方向是成立的。
3. 但其“自动继承工厂 ID”“单工厂单默认共享通道”“internal/external 成对槽位”都只应保留为迁移兼容层，不再新增依赖。
4. 自阶段 6 起，新的共享端点必须接入独立 Binding Core，通过 binding ID 建立点对点关系，而不是贴入口自动获得长期绑定身份。
5. 两类端点在未绑定状态下，仍应尽量分别沿用 Create Item Vault 与 Create Fluid Tank 的本地多方块行为和自动化语义。
6. 端点一旦绑定成功，方块实体本身不再把本地 inventory 或本地 tank 当作真实数据源；真实真相转移到 binding ID 对应的共享后端，本地控制器只保留绑定信息、局部结构信息、能力投影与必要的显示同步信息。
7. 共享流体罐绑定后仍应保持“单一流体类型”约束，不允许同一个 binding ID 的流体后端同时混入多种流体。

#### 5.4 存储后端设计原则

1. 共享库存数据必须脱离具体方块实体，统一保存在世界级持久化结构中。
2. 长期最小寻址键应采用 binding ID，而不是工厂 ID、玩家 UUID 或端点绝对坐标。
3. 工厂上下文、区块上下文与目标位置仅作为绑定元数据存在，不承担共享后端主键职责。
4. 共享物品箱后端与共享流体罐后端应分开建模，不应在第一版就尝试共用一套混合槽位结构。
5. 若阶段 5 兼容层仍存在旧通道键，也应只作为历史迁移数据，而不是新模型的正式 API。
6. 同一 binding ID 下的所有同类端点共享同一份后端真相，并共享同一份变更版本号；任何一次插入、抽取、合并、清空，都只修改后端主数据，并由端点按需读取结果。
7. 不允许为不同端点各自缓存一份完整物品列表或完整流体快照后再做松散同步，以免产生一致性和性能问题。
8. 共享存储只有在实际内容变化时才标脏保存，不应因为玩家靠近、界面打开或端点存在就持续写盘。
9. 若未绑定端点切换到已绑定状态，本地已有内容必须一次性并入共享后端，不再保留“本地留一份、后端再同步一份”的双真相路线。
10. 若绑定迁移失败，则整个绑定与迁移过程必须回滚，不能出现只绑定成功一半、只迁移成功一半的脏状态。
11. 后端必须区分“当前共享容量”和“当前真实存量”，不再把历史曾达到过的容量继续当作对外暴露容量。
12. 缩容后若真实存量超过当前共享容量，系统进入超容占用状态：内容不丢、不吐出，但对外写入规则必须受当前共享容量约束。
13. 容器类绑定一旦解绑或失效清算，后端状态必须随 binding ID 一起被销毁；物品类应直接掉落实体，液体类可直接清空，不应继续保留无访问路径的死数据。

#### 5.5 正式语义规则（2026-05-16）

1. 共享物品箱与共享流体罐都应被定义为“同一个共享语义容器在两个位置上的本地表现层”，而不是两个彼此复制内容的普通容器。
2. 对流体端点来说，任意两个绑定到同一 binding ID 的同类流体端点，无论本地尺寸差多少，都必须读取同一个共享总量与同一个共享总容量。
3. 共享流体罐的液位比例按“共享真实液量 / 当前共享总容量”计算；两侧本地罐体都按同一比例显示液位，而不是按各自本地尺寸单独计算。
4. 若共享真实液量超过当前共享总容量，则所有本地流体罐都显示为满，并禁止继续输入；只有消耗掉部分液体或重新扩容后，才允许继续输入。
5. 共享流体罐绑定后仍保持单流体约束；绑定迁移时若两侧流体类型不兼容，则必须拒绝绑定或要求先清理，不允许生成混合真相。
6. 蒸汽机、蒸汽喷口、锅炉热量评估及相关附属语义，必须按同一 binding ID 下所有有效共享流体端点对应的共享语义容量与共享真实液量共同计算，而不能只按本地单侧罐体计算。
7. 共享流体罐的本地控制器需要同时维护两类信息：一类是用于多方块外观与局部结构判断的本地几何尺寸，另一类是用于容量、液位、锅炉和蒸汽语义判断的共享语义容量。
8. 对物品端点来说，任意两个绑定到同一 binding ID 的同类物品端点，无论本地尺寸差多少，都必须暴露同一个共享物品处理器与同一个共享总容量。
9. 共享物品箱的当前共享容量由当前已绑定的内外保险库结构共同决定；若缩容后已有内容超过当前共享容量，则进入超容占用状态，但物品不会自动掉落。
10. 共享物品箱处于超容占用状态时，应允许继续向已有未满堆叠填充，但禁止任何需要额外新槽位的插入；提取始终允许。
11. 共享物品箱的对外暴露规则必须以“当前共享容量”为准，而不能把历史曾达到过的更大容量继续暴露给漏斗、机械臂或其他自动化端点。
12. 共享流体罐的对外暴露规则也必须以“当前共享容量”为准；历史容量只能作为后端曾经达到过的状态参考，不能继续作为当前可写入容量。

#### 5.6 性能边界

1. 阶段 5 首版禁止引入任何“每 tick 扫描附近入口/附近接口/附近网络”的常驻逻辑。
2. 绑定关系应在放置、拆除、邻接变化或玩家显式操作时离散更新，而不是持续轮询。
3. 共享接口块若无内容变化、无玩家交互、无自动化抽插请求，则不应主动触发额外更新。
4. 若后续接入多种自动化端点，依旧应通过统一的后端读写路径处理，避免不同接口各自维护同步线程或重复缓存。
5. 阶段 5 首版的性能目标是：接口数量增加后，额外开销主要体现在真实插拔操作次数和必要的显示同步上，而不是空闲端点数量上。

#### 5.7 玩家交互与基础能力

1. 共享接口块不新增标准箱式 GUI；该阶段的核心目标是稳定承接自动化输入输出，而不是玩家手动开仓界面。
2. 未绑定的共享物品箱应尽量保持 Create Item Vault 风格的本地自动化行为，未绑定的共享流体罐应尽量保持 Create Fluid Tank 风格的本地自动化行为。
3. 已绑定的共享物品箱必须继续完整支持漏斗、溜槽、机械臂、传送带端点等通过物品能力进行的插入与提取。
4. 已绑定的共享流体罐必须继续完整支持管道、管道泵以及其他通过流体能力进行的插入与提取。
5. 已绑定的共享流体罐必须继续正确显示共享液位比例、亮度与必要的控制器显示状态，不能因为后端化而退化成“只有数字变了、罐体不跟着变”的伪兼容。
6. 已绑定的共享流体罐应优先保持与锅炉、蒸汽机、蒸汽喷口等依赖本地罐控制器语义的方块兼容，而不是额外要求这些方块改为直接识别共享后端。
7. 第一版共享物品箱与共享流体罐都不区分输入端和输出端，所有已绑定端点默认都指向同一个 binding ID 下的对应共享目标。
8. 该条旧路线已经被阶段 6 修正：解绑后不应继续保留无访问路径的后端内容，而应执行 binding ID 级别的清算与释放。

#### 5.8 阶段 5 与阶段 5.5 的边界

1. 阶段 5 首版先验证“独立接口块 + 中心化共享存储后端”这条主链路。
2. 阶段 5.5 再讨论把当前单方块入口升级为多方块结构，并为共享接口预留专用结构位。
3. 到阶段 5.5 时，允许把现有独立接口块保留为外挂式兼容端点，或把其访问能力吸收到多方块入口结构中。
4. 但无论前端结构如何变化，阶段 5 建立的“绑定信息与库存后端分离”的设计原则不应回退。

#### 5.9 阶段 5 首版验收重点

1. 同一 binding ID 下的多个同类接口块必须显示完全一致的库存内容。
2. 玩家外部接口与工厂内部接口之间的共享库存必须实时一致，不允许出现端点副本漂移。
3. 历史阶段 5 原型中的自动绑定流程只作为过渡验证，不再作为阶段 6+ 的正式约束。
4. 未绑定接口、失效工厂、异常数据等情况必须有明确反馈，不能悄悄创建孤儿库存。
5. 阶段 5 首版完成后，传送带桥、管道桥、溜槽桥应能够明确复用同一后端，而无需重写存储模型。
6. 第一版应至少验证一个入口同时挂接共享物品箱端点和共享流体罐端点时，两者能各自稳定访问对应后端，而不会互相覆盖绑定信息。
7. 第一版应验证未绑定端点在本地容器模式和已绑定端点在共享模式之间切换时，不会出现吞物、吞液、复制或脏状态残留。
8. 第一版应验证共享流体罐在不同本地尺寸下仍能按同一共享比例显示液位，并在超容状态下显示为满且拒绝继续输入。
9. 第一版应验证共享流体罐与锅炉、蒸汽机、蒸汽喷口等依赖本地控制器语义的方块能够继续按共享语义容量工作。
10. 第一版应验证共享物品箱在缩容导致超容后，已有堆叠可继续补满，但任何需要新增槽位的插入都会被拒绝。

## 阶段 5.5：入口结构升级版

目标：

在不推翻阶段 5 共享存储后端的前提下，把当前单方块入口升级为更适合承载接口位的多方块结构。

内容：

1. 为入口结构预留专门的共享接口位置，而不是继续让传送入口承担全部交互。
2. 明确多方块入口的主方块、装饰方块、接口位和失效恢复规则。
3. 决定阶段 5 的独立接口块在新结构下是继续保留，还是降级为兼容扩展块。
4. 保持原有传送逻辑与共享后端语义原则不变，但绑定真值必须逐步迁到 binding ID 模型，只替换前端访问形态而不回退到工厂 ID 主键。

### 阶段 5.5 当前兼容性约束（2026-05-16）

基于对 Create、Simulated、Aeronautics 与 Sable 的前期源码阅读，阶段 5.5 的多方块入口结构先固化以下约束：

1. 多方块结构的绑定真相不能建立在“当前是否处于 Contraption 或 SubLevel 中”这一运行时状态上，而必须落到我们自己的持久化结构记录中。
2. 多方块结构识别应采用按需、有上限的结构扫描与校验流程，避免常驻维护整张结构成员图。
3. 强力胶、蜂蜜胶及类似胶接物品只适合作为 authoring 或辅助扫描语义，不适合作为运行时绑定的最终真相来源。
4. 入口主方块、共享接口主端点和工厂绑定信息应尽量保持为静态世界锚点，不应默认允许它们直接成为可移动物理结构的一部分。
5. 若玩家用 Create 或 Aeronautics 将入口附近结构物理化，系统应优先保证绑定不丢失，再决定是否允许该结构继续被识别为“有效入口外壳”。
6. 后续若要支持“物理化后恢复识别”，应优先基于 binding ID、结构类型、控制器位置、局部坐标和端点元数据重建识别，而不是依赖原世界绝对坐标。
7. 对 Create 的借鉴重点应放在 `Contraption` 一类按需 BFS、容量上限、visited/frontier 缓存等结构搜索策略上，而不是直接复用其物理化成员语义。
8. 对 Simulated/Aeronautics 的借鉴重点应放在装配与反装配生命周期钩子、局部坐标变换和运行时投影思路上，而不是直接把 SubLevel 作为我们的业务模型。
9. 在 Pocket Factory 维度内，Create/Simulated/Aeronautics 的动态结构原则上应被视为“允许存在的外部兼容对象”，但我们自己的入口、多方块绑定与共享接口系统不能对其成功装配或物理参数做硬依赖。

阶段 5.5 的直接设计含义：

1. 入口结构的“识别”和“运行时表现”必须拆开设计。
2. 结构识别结果必须能在静止状态下独立保存，并在后续重新扫描时恢复。
3. 共享接口如需兼容物理化环境，优先做外挂式兼容端点，而不是要求整个入口结构始终保持可移动。

### 阶段 5.5 当前共享原型前置约束（2026-05-16）

基于对 Create 传送带、漏斗、溜槽、流体管道、机械泵与便携式接口源码的前期阅读，后续共享原型先固化以下约束：

1. “关联带”不应被设计为真正跨维度延伸的 Belt 路径，而应被设计为两端各自留在本地世界中的 Belt 风格端点，中间仅通过共享层交换物品状态。
2. “关联漏斗”与“关联溜槽”不应共用同一套内部语义；漏斗更接近主动抽取与过滤端点，溜槽更接近单槽下落与交接端点，后续实现时应拆开。
3. “关联管”不应直接把两个维度中的 Create 流体网络拼接成一个统一网络，而应把每一侧都视为本地网络端点，通过共享流体后端或共享流量状态做桥接。
4. “关联泵”不应直接跨维度驱动对侧网络的压力计算或重建流程，而应只影响本侧端点的抽取/压出语义，由共享层完成跨维度流量结算。
5. 所有共享物品接口都必须显式区分 simulate 和 execute 两个阶段，避免两端在同一个 tick 内都认为自己成功而造成复制、吞物或顺序错乱。
6. 所有共享流体接口都应采用至少一步握手或完成态标记，避免半连接、半更新或临时反向流被另一侧当成稳定流向读取。
7. 共享层不得假定源和目标一定同时已加载；若目标维度、目标区块或目标端点当前不可用，应优先回退为“本侧暂不提交”，而不是先删本侧状态再等待远端恢复。
8. 若后续需要对接 Create 风格的便携式接口、移动结构或其他 keep-alive 连接模型，应把它们当成“局部在线判定”参考，而不是直接把实体引用或单世界距离逻辑搬到跨维度系统里。
9. 后续共享原型必须优先保证守恒、顺序和失败回退，再考虑视觉表现与传输速度拟真；只要这三项不稳，就不能继续扩展更多外部接口种类。

这组约束对后续原型的直接含义：

1. 第一版共享原型更适合从“关联漏斗/关联箱式输入输出”而不是“完整关联带/完整关联管网”开始。
2. 液体共享原型更适合先做“共享池 + 本地端点”模型，而不是直接仿真跨维度流体网络。
3. 若后续需要支持传送带观感，应先把它做成端点外观和本地交互协议，而不是立即追求带上物品跨维度连续可见。

## 阶段 6：自动化兼容版

目标：

将工厂接入现有自动化网络，但绑定关系不再以工厂为主键；阶段 6 的主目标改为先抽离以 binding ID 为核心的统一绑定模块，再以“关联溜槽”作为第一种真正可扩展的物流端点。

内容：

1. 阶段 6 首版先完成“绑定加强”，把当前共享箱罐与后续自动化端点都会复用的绑定逻辑抽离为独立模块。
2. 第一种吃这套新绑定模块的自动化端点，不再优先选择传送门，而改为“关联溜槽”。
3. 关联溜槽的职责是“本地表现为普通垂直溜槽，跨维度通过受控后端完成交接”，而不是“把掉落物实体传送到另一侧”。
4. 阶段 6 首版继续优先支持离散、可回退、易验证守恒的端点类型，再逐步扩展到更复杂的连续传输表现。
5. 旋转动力、应力耦合、完整传送带连续显示、复杂流体网络拼接都先保留为后续切片，不作为 6.0 首版目标。

### 阶段 6 当前方案（2026-05-17）

#### 6.1 总体路线调整

1. 已经实现过的“传送门 + 物品实体穿门”原型，只保留为验证阶段 6 可行性的探路结果，不再作为正式主路线继续扩展。（原因是大量掉落物导致的性能差，与本mod项目方向不符）
2. 阶段 6 的正式主路线改为“绑定加强先行，关联溜槽首发，其他端点后续复用”。
3. 本阶段不再继续放大“单工厂一个默认端点”的历史约束，而是把绑定关系正式抽象为独立模块。
4. 工厂 ID 继续保留为归属、恢复和显示上下文，但不再承担绑定主键、共享后端主键或唯一通道键职责。
5. 正式的绑定真相应升级为“binding ID + endpoint A + endpoint B + policy snapshot + 可选工厂上下文元数据”。
6. internal/external 角色只允许作为显示或策略层辅助概念保留，不再作为 Binding Core 的核心真值结构。

#### 6.2 绑定加强：统一绑定核心

1. 阶段 6 开始，应把当前共享物品保险库、共享流体储罐以及后续关联溜槽都会用到的绑定逻辑抽离为独立的 Binding Core，而不是继续让每种端点各自维护一套绑定流程。
2. Binding Core 的最小职责包括：端点注册、端点失效清理、点对点建立绑定、解绑、重绑、持久化、合法性校验与客户端选择提交。
3. Binding Core 的最小端点抽象至少应包含：端点类型、所在维度、控制位置、可选结构主控位置、endpoint key、可选工厂上下文、可选区块上下文、当前 binding ID 与最后一次校验结果。
4. 首版绑定关系建议只允许“同类端点对同类端点”建立点对点绑定，例如关联溜槽只绑定关联溜槽，关联物品保险库只绑定关联物品保险库。
5. 当前共享箱罐使用的“默认单通道”只可作为阶段 5 历史兼容层保留，但阶段 6 起不应再把它视为长期上限。
6. 一个工厂在后续必须允许存在多个物品绑定、多个流体绑定以及多条物流绑定，不再假设“巨大单工厂只靠一个共享物品箱或一个共享流体罐就足够”。
7. 每次新绑定都必须创建新的 binding ID；binding ID 不复用，旧 binding ID 只在解绑或失效清算时销毁。
8. Binding Core 还必须维护 endpoint index，用于把 endpoint key 快速映射到 binding ID，从而支持红框反馈、失效清理、重绑校验与服务端 O(1) 占用判断。
9. 绑定加强的直接目标不是一次性做完复杂权限系统，而是先把“binding ID 独立真值”“多绑定并存”“多类端点共用一套绑定基础设施”稳定下来。

#### 6.2.1 默认维度策略与可配置放宽

1. 新 Binding Core 默认要求每条绑定至少有一个端点位于口袋工厂维度。
2. 新 Binding Core 默认要求两个端点位于不同维度。
3. 这两条规则必须做成服务端配置项，而不是硬编码在核心物品或单个端点适配器里。
4. 推荐至少提供两个独立配置：`requirePocketFactoryEndpoint` 与 `requireDifferentDimensions`。
5. 默认组合应为 `true + true`，即“一个口袋工厂端点 + 一个其他维度端点”。
6. 若关闭 `requireDifferentDimensions`，则允许“一个口袋工厂端点 + 一个同维端点”。
7. 若两条都关闭，则系统进入完全通用关联模式，允许任何维度、包括同维建立绑定。
8. Binding Record 在创建时应保存一次 policy snapshot，用于调试、兼容未来规则迁移和可解释性反馈。

#### 6.3 玩家交互与目标选择

1. 玩家交互应继续复用当前口袋工厂核心作为绑定工具，而不是为每一种端点再发明一套新物品或新菜单。
2. 绑定操作应从“按工厂默认通道绑定”升级为“先选端点 A，再选端点 B，服务端校验通过后创建新的 binding ID”。
3. 核心物品中的临时状态应保存“已选择端点引用”和必要的绑定模式，而不是把中间状态写在普通方块物品或临时掉落物上。
4. 工厂内部端点仍应优先通过工厂维度位置推断所属工厂；但该工厂上下文只参与显示、默认策略校验和恢复辅助，不再作为绑定寻址键。
5. 客户端应加入 Create 风格的目标选择反馈：可绑定目标高亮、已选源端点持续描边、非法目标给出明确失败提示。
6. 这套视觉反馈从阶段 6 首版架构就应被纳入，而不是等功能做完后再补；因为它本身就是“多绑定通道”可用性的组成部分。
7. 入口不再被规定为“绑定提交点”；在新模型里，它可以是普通候选端点，也可以仅作为进入工厂的设施存在。

#### 6.4 关联溜槽的产品定位

1. 阶段 6 第一种正式自动化端点确定为“关联溜槽”，而不是继续扩展传送门。
2. 关联溜槽应被定义为“本地表现为普通垂直溜槽、跨维度通过 binding ID 对应的桥接后端完成交接的溜槽端点”，而不是“把溜槽掉出的实体在另一侧重新生成”。
3. 阶段 6 首版只做普通垂直溜槽，不做智能过滤，不做普通版以外的智能溜槽增强线。
4. 首版不要求立刻完成斜向输入输出、窗口外观、包裹实体专用表现和鼓风机上吹语义，但结构上必须预留这些扩展点，避免后续重做状态格式。
5. 关联漏斗与关联溜槽继续保持分离；漏斗更接近主动抽取与过滤端点，溜槽更接近单槽缓存、下落推进与端点交接，不应共用同一套内部状态机。

#### 6.5 关联溜槽的行为原则

1. 关联溜槽不应把“掉落物实体跨维度传送”当作主路径；主路径必须改为溜槽内部单栈运输状态在两端之间受控交接。
2. 关联溜槽应尽量学习普通 Create 溜槽的运输模型：本地保存当前 ItemStack 与推进位置，向上下游交接时优先使用本地能力与溜槽语义，只有最终兜底时才允许掉落实体。
3. 当 A 端到达输出阶段时，不是让物品直接从 A 端漏出，而是先尝试写入绑定后的 B 端接收缓冲；B 端再把它当作“刚进入本溜槽的内容”继续向下推进。
4. 若目标端未加载、目标端忙碌、目标端失效或本次桥接校验失败，源端本次提交必须留在本侧等待或回退，不允许先删本侧状态再等远端恢复。
5. 关联溜槽首版优先支持垂直向下语义；后续若需要支持风扇上吹，应在同一运输骨架上扩展向上路径，而不是推翻首版数据模型。
6. 未绑定的关联溜槽若实现成本可控，应尽量保持普通垂直溜槽风格的本地行为；若首版不能完整复刻，也必须保证失败模式明确，不制造吞物或伪兼容。

#### 6.6 阶段 6 的数据与职责分层

1. 世界级持久化层应分开保存三类真值：Binding Record、Endpoint Index、以及各端点类型自己的共享后端或桥接后端状态。
2. 对共享箱罐来说，真实真相仍然是 binding ID 对应的共享库存或共享流体；对关联溜槽来说，真实真相不应被设计成“大共享库存”，而应是 binding ID 对应的配对关系与必要的跨维度交接缓冲。
3. 方块实体应继续只保存少量本地状态：端点类型、当前 binding ID、工厂上下文、本地显示缓存、局部结构信息与必要的运行态；不应把跨维度真值散落在各个 BE 里。
4. 建议把阶段 6 的实现明确拆成三层：Binding Core、Endpoint Adapters、Binding Tool UX。
5. Binding Core 只负责端点注册、binding ID 分配、binding record、endpoint index、策略校验与失效清理；Endpoint Adapters 负责把共享箱罐、关联溜槽等各自接入统一端点抽象；Binding Tool UX 负责核心物品、选择态、高亮、提交与反馈。
6. Binding Record 建议最少包含：binding ID、binding kind、endpoint A、endpoint B、policy snapshot、创建时间、可选调试标签。
7. Endpoint Ref 建议最少包含：endpoint type、dimension、pos、endpoint key、可选工厂上下文、可选区块上下文、可选结构主控位置。

#### 6.7 推荐实现顺序

1. 第一优先级：重写 Binding Core 真值层，把现有“工厂 -> 通道 -> internal/external”结构迁成“binding ID -> binding record + endpoint index”。
2. 第二优先级：把现有共享物品保险库与共享流体储罐迁到这套统一绑定基础设施上，确保阶段 5 的已有功能不被遗忘在旧模型里。
3. 第三优先级：为核心物品加入统一的端点选择、点对点提交和基础高亮反馈，完成“绑定加强”的最小可用闭环。
4. 第四优先级：实现关联溜槽首版，只支持普通垂直版，并以它作为 Binding Core 的第一种真正自动化端点消费者。
5. 第五优先级：在关联溜槽稳定后，再考虑关联漏斗、传送带末端端点、流体桥接端点等其他适配器。
6. 第六优先级：等端点语义稳定后，再补足窗口外观、包裹实体显示、鼓风机上吹、更多目标特效和更完整的调试信息。

#### 6.8 同步、守恒与性能原则

1. 阶段 6 继续坚持“服务端后端为唯一真值，客户端只保留显示与交互镜像”的原则。
2. 所有绑定关系和桥接提交都必须显式区分 simulate 与 execute；只要目标端不可提交，就不得在源端先行销毁本地状态。
3. 关联溜槽、共享箱罐与后续其他端点都不允许引入“按端点常驻高频轮询整个网络”的空转开销。
4. 一个 endpoint key 默认只允许属于一个 binding ID；除非未来明确引入多路复用设计，否则不应允许一个端点同时参与多个绑定。
5. 容器类 binding 一旦解绑或失效清算，必须立即释放对应 binding ID 的后端状态，避免遗留无访问路径的死数据。
4. 只有内容变化、绑定变化、连接变化或玩家可见且需要刷新的交互场景，才应触发同步与显示更新。
5. 阶段 6 首版的性能目标应从“空闲端点数量几乎不增加额外负担，主要开销集中在真实物流提交与必要显示同步”出发，而不是继续依赖大量掉落实体维持跨维度物流。
6. 调试信息可以显示绑定键、端点类型、工厂上下文、当前连接状态和最近一次提交结果，但这些信息不应反向成为业务真值。

风险：

1. 不同模组的能力系统差异较大。
2. 若继续把单工厂单默认通道当成长期模型，后续一旦加入多条物流线、多个共享容器或多类端点，重构成本会明显上升。
3. 若直接拼接不同位面的原生传送带、管道或机械网络，复杂度和调试成本都会显著上升。
4. 直接绑定具体方块行为会提高维护成本。
5. 应优先依赖抽象能力层，而不是大量硬编码兼容。

## 阶段 6 收尾记录（2026-05-17）

当前“关联溜槽”这一轮已经基本完成收口，可作为后续自动化端点继续扩展的稳定基线。

### 已完成的关联溜槽能力

1. 绑定事实源已经完全收敛为“普通垂直溜槽 + 手持核心手动绑定”，不再依赖“处于工厂内部”或放置时自动推导绑定。
2. 关联溜槽已经成为隐藏实现物：无创造栏暴露、掉落普通溜槽、未绑定时尽量回退为普通溜槽。
3. 两端点之间已经具备稳定的单槽桥接后端，并完成了物品守恒与销毁回退。
4. 关联溜槽已补齐大部分原版普通溜槽手感，包括窗口/包铁/斜接外观、空手交互、侧壁放置形成斜溜槽、顶面掉落物吸入等。
5. 风扇联动第一版已完成，当前目标语义为“关联端点作为跨端接头，把本地下半段 push 和本地上半段 pull 续接到对侧网络”。

### 本轮验证后值得抽出的公共模块

本轮实现已经暴露出一批与“关联物品保险库 / 关联流体储罐 / 关联溜槽”共同拥有的模式，后续应收敛为独立模块，而不是继续散落在各端点类中。

1. 端点身份与定位模块。
当前 `LinkedStorageBindingHelper.endpointKey/parseEndpointKey` 已同时服务 storage 与 chute，后续应升级为更中性的“LinkedEndpointHelper”或“BindingEndpointHelper”，统一承载：
	1. endpointKey 编解码。
	2. 对端定位解析。
	3. 结构/端点基础遍历工具。
2. 端点恢复与回退保护模块。
当前 storage 有 `COLLAPSING_POSITIONS/PENDING_TRIGGER_REMOVALS`，chute 有 `RESTORING_ENDPOINTS`，它们本质上都是“端点回退过程中抑制递归清理”的保护机制，后续应抽成统一的 restore/collapse guard。
3. 手动绑定交互模块。
当前核心物品在 storage 与 chute 上都已经形成了“hover 可选态 / click 两段式选择 / validate pair / create binding / rollback on second failure”的统一交互骨架，后续应该下沉一层，避免在 `PocketFactoryCoreItem` 中继续复制一套新端点逻辑。
4. 关联端点生命周期模块。
storage 的 orphaned normalize、chute 的 orphaned normalize、远端恢复、解绑后回退，本质都是“绑定失效后的自修复/自回退”，后续应提炼成统一生命周期钩子：绑定建立、绑定失效、端点卸载后加载、对端摧毁后的恢复。

### 本轮试错后可清理的冗余与无用代码

以下内容不是必须立刻删除，但已经被确认是过渡性代码或风险较高的试错残留，下一轮适合开始清理。

1. `LinkedChuteBlockItem` 的玩家语义仍偏重。
当前它虽然已经隐藏，但仍保留了一整套 bindingId/factoryId tooltip 和 bound stack 语义。若后续产品规则继续坚持“隐藏实现物”，它应逐步退化为：
	1. 仅服务极少数内部恢复/调试用途。
	2. 不再承载玩家可感知说明。
2. `PocketFactorySavedData.getChuteSnapshot(...)` 目前主要是调试时代的桥接观察接口。
若后续没有持续使用，可降级为调试专用接口或删除，避免误导为正式公共 API。
3. `LinkedChuteBlockEntity` 中的反射字段访问目前是必要过渡方案，但仍属高脆弱点。
当前 `itemPosition/pull/push/updateAirFlow` 的反射访问说明 chute 行为是通过“外挂补丁”接入的。若后续端点类型继续增加，应考虑是否有必要把一部分共用控制封装成显式适配层，减少多处直接反射复制。
4. 关联溜槽顶面掉落物吸入兜底逻辑属于“兼容修复补丁”而不是结构优雅方案。
若后续确认原版入口路径在 hidden linked chute 上已经完全稳定，应重新评估是否可将该兜底缩到更窄，避免未来重复吸取或双路径维护。

### 本轮经验结论

1. 对于“隐藏实现物”路线，最重要的不是先把后端做大，而是先把绑定事实源、掉落、放置、销毁回退、hover/click 这些产品语义收干净。
2. 共享后端类端点（物品保险库、流体储罐）与桥接后端类端点（溜槽）虽然传输模型不同，但绑定、回退、孤儿自修复、对端恢复、核心交互这些壳层逻辑高度相似，应尽快去重。
3. 溜槽证明了“跨端延续原版局部网络语义”是可行路线，后续端点应优先保持原版玩法手感，只在端点处替换掉“目标端”而非重做整个系统。

### 下一步建议：先做关联动力泵，再决定是否回到关联管道

当前更推荐下一步优先做“关联动力泵”，而不是继续直接攻关联管道，也不是直接进入关联传送带。

原因：

1. 动力泵天然就是 Create 流体系统中的方向控制器与压力源，产品语义比关联管道清晰得多。
2. 关联管道要求把“跨维桥接”伪装成“局部连续管网”，这会同时碰到 pressure、flow、endpoint、open end 与网络生命周期多层耦合，当前验证已证明这条路极不稳定。
3. 关联动力泵只需要让“两台泵之间的流体传输发生交叉”，而不是伪造整段跨维连续管网；这更接近关联溜槽的成功经验：隐藏实现物 + 明确方向 + 局部续接。
4. 传送带仍然比流体系统更重，先做关联动力泵能继续验证 Binding Core 对“第二类桥接后端”的承载能力，同时避开关联管道目前最难的那层图连通性伪装。

建议的下一阶段顺序：

1. 先做关联动力泵原型，优先完成“成对泵跨维抽放”的最小闭环。
2. 在关联动力泵落地时同步沉淀 bridge backend 的方向、状态机、握手与回退规则。
3. 待关联动力泵稳定后，再判断是否仍有必要继续挑战关联管道；若继续做，应把关联动力泵沉淀出的桥接层复用过去。
4. 关联传送带保持在关联动力泵之后，不提前展开。

### 关联动力泵设计目标（阶段 6 新方向）

关联动力泵对应原版 Create 的动力泵，但两台绑定到同一 binding ID 的关联动力泵之间会形成跨维流体交叉。

产品目标：

1. 玩家对单台关联动力泵的直觉应几乎等同于原版动力泵：它仍然有明确朝向、仍然依赖转速、仍然通过前后两侧控制抽取与释放。
2. 唯一新增语义是：若两台关联动力泵已绑定，则泵本应连接的“另一侧本地流体网络”会被另一台关联动力泵的对应侧替换。
3. 对玩家来说，它更像“一对跨维同步的动力泵”，而不是“共享储罐”或“共享管道块”。
4. 它应与关联溜槽保持同一产品风格：隐藏实现物、手动绑定、原版局部手感尽量不变，只有两个关联端点之间发生交叉。

#### 行为一致性约束

第一版必须尽量保持以下原版动力泵行为：

1. 泵是否工作仍由本地转速决定；没有动力时不应跨维传液。
2. 泵的抽/放方向仍由原版前后朝向与正反转逻辑决定，不新增额外的 UI 模式切换。
3. 泵对本地网络施加压力的方式应尽量复用原版路径，而不是外挂另一套平行 pressure 系统。
4. 两台关联动力泵都存在且绑定有效时，跨维交叉只发生在“泵的远端”这一层；泵两侧的本地普通管网仍按原版处理。
5. 若对端不存在、未加载、失效、解绑或方向不兼容，本端应尽量回退到原版动力泵语义，而不是卡死在半连接状态。

#### 与关联管道的语义差异

关联动力泵不是“被动延长的跨维管道”，而是“主动跨维抽放设备”。

必须明确：

1. 它不承诺把两个维度中的普通管网拼成一张统一连续图。
2. 它只承诺把 A 泵的抽取结果，通过 binding ID 定向交给 B 泵对应的释放侧，反之亦然。
3. 因为原版动力泵本来就承担 pressure source 角色，所以这条产品语义比关联管道更贴近 Create 的原生抽象。

#### 绑定与状态机

第一版建议新增独立通道类型，例如 `LINKED_PUMP`，不要继续复用 `LINKED_PIPE`。

绑定建议：

1. 仍由 Pocket Factory Core 手动选择两台动力泵并建立 binding ID。
2. 每台关联动力泵仍记录 bindingId、factoryId、internal/external endpoint 等统一字段，保持与现有 Binding Core 一致。
3. 绑定成功后，两端不共享“库存真相”，只共享“泵对泵交叉关系”和必要的瞬时桥接缓冲。

运行态建议至少区分以下状态：

1. Idle：未绑定、无动力、对端不可用或当前不满足抽放条件。
2. PullingRemote：本端按原版语义正在本地抽取，并把结果送往对端泵的释放侧。
3. PushingRemote：本端正在把对端泵送来的流体按原版释放到本地网络。
4. FallbackLocal：绑定虽存在，但当前对端无效或握手失败，本端退回原版动力泵行为。

#### 桥接后端约束

第一版仍允许存在小型瞬时桥接缓冲，但它只能作为内部交接媒介，不能暴露成玩家可见的共享储罐语义。

约束如下：

1. 缓冲只用于跨维 tick 时序解耦与守恒，不应被本地普通泵或普通管网当作独立储液源直接读取。
2. 任一时刻只能存在一个当前有效交付方向，避免双向同时写入同一缓冲造成混向。
3. 若本端持续抽取而对端不可释放，应像原版泵遇到终点堵塞一样逐步停止有效传输，而不是无限吞入。
4. 桥接缓冲的类型必须保持单流体约束，不允许在同一 binding 中混入不同流体。

#### 第一版最小可测切片

第一版不要追求完整兼容所有原版细节，先做以下最小闭环：

1. 两台已绑定关联动力泵，分别连接在两个维度中的普通流体管网里。
2. A 泵本地前侧接液源，后侧接普通管网；B 泵本地释放侧接普通管网和流体储罐。
3. 当 A 泵按原版语义抽取时，B 泵按原版释放侧方向把流体稳定送入本地普通管网或储罐。
4. 翻转任一端转向或改变转速后，应表现出与原版动力泵尽量一致的抽放切换，不应出现多方向同时释放。
5. 对端被拆除、解绑或未加载后，本端应恢复为原版本地泵行为或安全停机，不能凭空生成水或残留假流。

#### 第一版验收重点

1. 不出现“可见流动但不真实输液”的假流。
2. 不出现对空气错误放水、开放端误出液或双向同时释放。
3. 关联动力泵不退化为共享储罐，也不要求玩家在 B 侧额外再放一个普通泵才能工作。
4. 玩家在大多数场景下可以直接按“原版动力泵的使用习惯”理解和使用它。

### 阶段 6 收尾记录（2026-05-18）

当前“流体传输”环节已经完成收口，可作为后续自动化扩展的稳定基线。

本轮收口后的明确结论：

1. 关联管道路线已经终止，不再保留实现、注册、兼容层或用户入口；阶段 6 的流体主线仅保留关联动力泵。
2. 关联动力泵当前采用“远端 source 接面 + 本地 bridge 缓冲”的混合方案，目标是尽量保持与原版动力泵一致的玩法语义，只在绑定的两台泵之间发生跨端流体交叉。
3. 为降低复杂管网下的重复开销，当前实现已经加入保守缓存与加载后刷新逻辑，但桥接缓存本身不再跨重进世界持久化，避免旧运行态残留影响下一次进入。
4. 关联动力泵的客户端可视化已补齐到与原版尽量一致：除方块模型外，也已补上原版动力泵齿轮所依赖的 renderer 与 visual 路径。

阶段 6 收尾后的产品边界：

1. 当前已确认可继续向后推进的流体自动化能力只有“成对绑定的关联动力泵跨端抽放”。
2. 当前不再继续挑战“把两侧普通流体管网伪装成一张跨维连续管网”的关联管道方案。
3. 若后续继续优化流体性能，应优先围绕关联动力泵做局部节流、缓存与加载态修复，而不是重新开启关联管道路线。
4. 后续如进入新的自动化阶段，应把关联动力泵视为阶段 6 的稳定完成项，而不是待返工原型。


## 阶段 7：迁移与蓝图版

目标：

降低玩家把现有机器迁入工厂的成本。

内容：

1. 使用框选工具选择外部结构。
2. 将结构保存为蓝图或打包数据。
3. 在工厂内指定区域部署。
4. 后续再逐步支持真实方块实体与复杂网络恢复。

建议：

1. 先做蓝图保存与投影。
2. 再做有限兼容的实物搬迁。
3. 不要一开始就承诺完整兼容所有机械动力网络。

### 阶段 7 当前可学习说明（2026-05-16）

基于对 Sable 源码的前期阅读，当前对“结构迁移”最值得学习的点可以先简化归纳为以下几条：

1. 重点不是物理系统本身，而是把大结构搬运拆成固定流水线：收集结构、校验规则、坐标变换、搬运方块与方块实体、搬运附属对象、通知更新、清理旧位置。
2. 高性能关键不是常驻维护整张结构图，而是在需要搬迁时做一次有上限的结构搜索。
3. 结构搜索、连接判定、坐标变换、实际搬运、世界通知这些职责应拆开，而不是混在一个交互逻辑里。
4. 对我们最值得直接借鉴的是：一次性 gather、容量上限、可插拔连接判定、独立 transform、分阶段执行。
5. 不应直接照搬整个 sub-level 和物理系统本体；对 Pocket Factory 来说，这部分目前过重。

本轮前期归纳重点源码位置：

1. 整体装配入口：`modmake/docs/sable-main/common/src/main/java/dev/ryanhcode/sable/api/SubLevelAssemblyHelper.java` 中的 `assembleBlocks(...)`。
2. 有界结构收集：同文件中的 `gatherConnectedBlocks(...)`。
3. 附属对象随迁：同文件中的 `moveOtherStuff(...)` 与 `moveTrackingPoints(...)`。
4. 方块与方块实体搬运：同文件中的 `moveBlocks(...)`。
5. 世界更新通知：同文件中的 `markAndNotifyBlock(...)`。
6. 坐标变换抽象：同文件中的 `AssemblyTransform`。
7. 运行时承载层：`modmake/docs/sable-main/common/src/main/java/dev/ryanhcode/sable/sublevel/ServerSubLevel.java`。
8. sub-level 容器层：`modmake/docs/sable-main/common/src/main/java/dev/ryanhcode/sable/api/sublevel/ServerSubLevelContainer.java`。
9. plot chunk 承载与包围盒维护：`modmake/docs/sable-main/common/src/main/java/dev/ryanhcode/sable/sublevel/plot/PlotChunkHolder.java`。

对 Pocket Factory 的直接启发：

1. 后续若做“外部机器迁入口袋工厂”，应先实现自己的集中式迁移 helper，而不是把扫描和搬运逻辑散落到多个方块或物品中。
2. 应先做蓝图/投影和受限实物搬迁，再考虑更重的动态结构或完整物理语义。

## 7. 推荐的数据模型

以下仅用于早期规划，不代表最终代码结构，但建议尽早按此方向组织。

### 7.1 Pocket Factory

建议字段：

1. 工厂 ID。
2. 所有者 UUID。
3. 工厂名称。
4. 工厂维度位置。
5. 工厂尺寸。
6. 入口方块位置。
7. 是否启用常加载。
8. 创建时间。
9. 当前接口配置。

### 7.2 Player Return Record

建议字段：

1. 玩家 UUID。
2. 进入前维度。
3. 进入前坐标。
4. 进入前朝向。
5. 最近绑定工厂 ID。
6. 记录有效性标记。

### 7.3 Shared Storage Bridge

建议字段：

1. 工厂 ID。
2. 通道键。
3. 库存槽数据。
4. 变更版本号或脏标记。
5. 访问端点列表或端点索引。
6. 输入输出模式。
7. 最近一次绑定校验结果。

## 8. 风险清单

### 8.1 第一阶段核心风险

1. 传送后玩家卡墙、坠落或落点无效。
2. 返回点失效导致玩家被困。
3. 常加载在服务器重启后失效。
4. 工厂维度规则未生效，仍然刷怪。
5. 光照实现与现有渲染或光照相关模组表现冲突。

### 8.2 中后期风险

1. 多工厂实例会扩大存档和映射复杂度。
2. 跨维度库存桥涉及同步与一致性问题。
3. Create 生态专用接口兼容面广，维护成本高。
4. 结构迁移会牵涉方块实体 NBT、连接关系和朝向恢复。

## 9. 第一阶段建议测试清单

1. 在主世界不同位置使用入口物品，确认进入工厂成功。
2. 在工厂内使用入口物品，确认回到进入前原位。
3. 故意销毁或改变原坐标可站立性，确认能回退到出生点。
4. 服务器重启后再次测试传送与回传。
5. 在工厂放置持续运行的 Create 机器，离开后确认仍运作。
6. 检查工厂维度是否自然刷怪。
7. 检查光照设置关闭与开启时的表现。
8. 使用 spark 对比迁移前后玩家常驻区域的负载变化。

## 10. 当前开发建议

正式编码时，建议严格按以下顺序推进：

1. 先完成维度与传送闭环。
2. 再完成工厂盒子生成与安全落点。
3. 再完成回传记录持久化。
4. 再完成常加载和禁怪规则。
5. 最后补充贴图、配置项和体验细节。

不要在第一阶段内加入以下内容：

1. 多工厂实例。
2. 单方块入口以外的新入口形式。
3. 共享箱子桥。
4. Create 传送带/管道/溜槽兼容。
5. 结构打包迁移。

## 11. 里程碑结论

只要第一阶段能稳定完成，本项目就已经具备继续投入开发的价值。

判断标准不是“功能是否豪华”，而是：

1. 玩家是否愿意把机器搬进去。
2. 玩家常驻区域的帧率是否明显改善。
3. 工厂是否足够稳定，不会因为传送、重登或常加载出问题。

若上述三点成立，则第二阶段开始建设固定入口和实例化系统；若第一阶段收益不明显，则应优先复盘性能瓶颈，而不是继续扩展功能面。
